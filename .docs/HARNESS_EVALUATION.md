## Section 1: Harness Architecture & Planning

### 1.1 Architecture Design

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 1.1.1 | A harness architecture pattern has been explicitly chosen | ✅ | Intent-driven hierarchical harness; engines per domain |
| 1.1.2 | The choice is documented and justified for the clinical use case | ⚠️ | CLAUDE.md covers structure but not the *why* behind pattern choices |
| 1.1.3 | The system uses a specialized harness (fixed phases) | ✅ | Not general-purpose; each path is deterministically defined per intent |
| 1.1.4 | The harness is distinct from the LLM | ✅ | Orchestrator wraps agents; `_build_return()` contract enforced at every engine boundary |
| 1.1.5 | Different models are used for different roles | ✅ | Sonnet for planner, Haiku for extractor/output, fast model for intent classification |

**Score: 2/3**

**Current state:** Intent-driven hierarchical harness with correct model selection by role. Architecture is sound but the justification for choosing a specialized harness over a general-purpose one is not written down anywhere.

**Ideal state:** A purpose-built specialized harness with clearly defined phases, where no clinical step is left to the LLM's discretion about whether to execute it or not.

**Gap / Action items:** Write a one-page architecture decision record (ADR) documenting why specialized > autonomous for this clinical context.

### 1.2 Planning & Phase Definition

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 1.2.1 | The workflow is broken into clearly numbered, named phases | ⚠️ | Phases exist implicitly in orchestrator routing; never explicitly named or numbered |
| 1.2.2 | Phases are fixed (not dynamically generated by the LLM) | ⚠️ | Standard path: fixed. Planned path (`filtered_discovery`): LLM generates the execution plan |
| 1.2.3 | Each phase has a defined input, output, and success condition | ⚠️ | `_build_return()` contract defines output shape; success conditions implied, not declared |
| 1.2.4 | A plan/progress tracker is visible during execution | ❌ | SSE status events stream but no phase-level progress object is tracked |
| 1.2.5 | The LLM cannot skip, reorder, or add phases without human approval | ⚠️ | True for standard path; planned path lets LLM define execution steps |
| 1.2.6 | Phase transitions are gated | ⚠️ | Two gates exist (device resolution, clinical completeness); no general phase-gate pattern |

**Score: 1/3**

**Current state:** The standard routing path (most queries) is effectively fixed — intent classification → extraction → engine → output. The planned path (`filtered_discovery` intent) is the exception: the query planner LLM generates the execution plan including what steps to run, in what order, with what dependencies.

**Ideal state:** A strict sequential phase plan where each transition is gated and logged. For clinical queries specifically: 1. Intake & validation → 2. Guideline retrieval → 3. Recommendation matching → 4. Risk flagging → 5. Output generation.

**Gap / Action items:**
- Name and number the phases explicitly in code (even as constants)
- Gate the planned path so it cannot include clinical engine steps without a pre-defined template
- Add a phase-level progress object to SSE streaming

## Section 2: Sub-Agents & Context Isolation

### 2.1 Sub-Agent Design

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 2.1.1 | Complex tasks are delegated to sub-agents | ✅ | Chain, database, vector, clinical — each is a separate engine with a single responsibility |
| 2.1.2 | Each sub-agent is initialized with a fresh context window | ✅ | `base_agent.py` initializes fresh per call; conversation history injected explicitly |
| 2.1.3 | The orchestrator explicitly controls context injection | ✅ | Each engine receives only what the orchestrator passes via `input_data` |
| 2.1.4 | Sub-agents are scoped to a narrow, well-defined task | ✅ | Intent classifier, extractor, planner, output — each has one job |
| 2.1.5 | Independent sub-agents can run in parallel | ⚠️ | Intent + equipment extraction parallelized; planned path has topological sort; most standard paths sequential |
| 2.1.6 | Sub-agent outputs are structured data, not free-form text | ✅ | `_build_return()` enforced; JSON schema via Pydantic models |

**Score: 2/3**

**Current state:** Sub-agent isolation is well-implemented. Fresh context per agent, orchestrator-controlled injection, structured return contract. More parallel execution is possible in standard paths but not currently utilized.

**Ideal state:** An orchestrator that spawns isolated sub-agents per clinical reasoning task, collects structured results, and synthesizes them — never passing one sub-agent's output directly to another without orchestrator review.

**Gap / Action items:** Identify additional steps in standard paths that could run in parallel (e.g., session state load + intent classification + input rewriting).

### 2.2 Context Management

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 2.2.1 | The main orchestrator agent's context is kept lean | ⚠️ | Context assembled per agent but full conversation history passed without pruning |
| 2.2.2 | Large tool call outputs saved to files; only summaries passed to agent | ❌ | Full device database loaded into memory and passed directly; vector results passed in full |
| 2.2.3 | A context compaction / summarization mechanism exists | ❌ | No compaction mechanism; conversation history grows unbounded within a session |
| 2.2.4 | Token usage is monitored per session and per phase | ✅ | `token_usage` dict tracks per-agent call; synced to Firebase |
| 2.2.5 | Maximum context budget defined; graceful degradation when approaching it | ❌ | No budget ceiling; no degradation path when approaching context limits |

**Score: 1/3**

**Current state:** Token usage is tracked but not acted upon. Full conversation history and full database contents are passed to agents without pruning or summarization. This is the system's biggest technical debt — long sessions or large databases will silently degrade model performance with no protection.

**Ideal state:** Context is surgically managed. The orchestrator holds only a summary of each completed phase. Guideline text, retrieved documents, and large outputs live in the file system and are referenced, not inlined.

**Gap / Action items:**
- Add a sliding window or summarization step for conversation history (e.g., summarize turns older than N)
- Define a max token budget per agent call; fail explicitly rather than silently when exceeded
- Move large reference content (database records, vector results) to file-based references rather than inlining

## Section 3: Validation Loops & Reliability

### 3.1 Output Validation

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 3.1.1 | All LLM outputs used in downstream logic are validated against a structured schema | ✅ | Pydantic models for clinical variables; JSON schema validation in `base_agent.py` |
| 3.1.2 | Schema validation failures trigger a retry or escalation — not silent failure | ❌ | Fallback to `{"raw_text": content}` on parse failure — silent, downstream logic proceeds with empty data |
| 3.1.3 | Medical-specific outputs validated against controlled vocabulary | ⚠️ | COR/LOE values are enum-validated; guideline trial names not checked against a canonical list |
| 3.1.4 | Hallucination guard: citations verified before included in output | ⚠️ | MatchingEngine verifies recommendations exist in `enriched_recommendations.json`; vector search citations not verified post-retrieval |
| 3.1.5 | Outputs compared against retrieved guideline before surfacing | ⚠️ | Matching engine cross-checks structured DB; output agent has no rubric check against the source |
| 3.1.6 | A secondary LLM or deterministic check cross-validates before reaching doctor | ❌ | No secondary validation pass |

**Score: 1/3**

**Current state:** Pydantic validation exists at the input layer (clinical variable parsing). The silent fallback on JSON parse failure in `base_agent.py` is the critical gap — if the extractor agent returns malformed JSON during a clinical assessment, the system proceeds with empty structured data rather than halting.

**Ideal state:** Every phase output passes through a validation layer. Failed validations loop back automatically, with a max retry count before escalating to a human reviewer.

**Gap / Action items:**
- Replace silent `{"raw_text": content}` fallback with an explicit retry (max 1–2 attempts) then hard error for clinical paths
- Add a post-output check: does the recommendation's COR/LOE exist in `enriched_recommendations.json`?
- Log all validation failures to Firebase for trend analysis

### 3.2 Reliability Engineering

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 3.2.1 | Per-step success rates are tracked and measured over time | ❌ | Token usage tracked; pass/fail rates are not |
| 3.2.2 | Target SLO defined and current baseline measured | ❌ | No SLO exists; reliability is assessed via manual regression tests only |
| 3.2.3 | Agentic failure modes are catalogued | ⚠️ | Past failures documented in dev_log and memory; no formal failure mode taxonomy |
| 3.2.4 | Retry logic exists at the phase level | ❌ | Single-pass architecture; no retry at any phase |
| 3.2.5 | System fails safely — errors surface rather than returning partial/incorrect output | ✅ | `_build_return(status="error")` returns explicit error; incomplete runs do not produce partial recommendations |
| 3.2.6 | Human-in-the-loop gate before recommendation reaches clinician | ❌ | System is real-time streaming; no review gate between engine output and clinician view |
| 3.2.7 | System can resume mid-workflow from a saved phase checkpoint | ❌ | Session state persisted in Firebase but no phase-level checkpoint; crash loses that turn's in-progress work |

**Score: 1/3**

**Current state:** The system fails safely when an error is caught (`status="error"`). The critical gaps are: no human review gate (clinical reasoning reaches clinicians in real time with no opportunity to intercept), and no success rate tracking (degradation is invisible until a clinician reports a bad recommendation).

**Ideal state:** The system treats each phase like a unit of software with a measurable pass rate. Reliability is tracked. No clinical output ever reaches a doctor without passing a human review gate.

**Gap / Action items:**
- Add a "reviewed by clinician before acting" disclaimer enforced in every clinical response (minimum viable gate)
- Log phase-level pass/fail to Firebase alongside token usage (single boolean per phase per turn)
- Add one retry with corrective prompt on LLM step failures before escalating

| 3.3.5 | System documents which steps are deterministic vs. LLM-driven | ✅ | CLAUDE.md explicitly documents the Python/LLM boundary and the test for each |

**Score: 2/3**

**Ideal state:** Clinical logic that can be codified is in Python. The LLM handles reasoning and language. The boundary is documented and enforced.

**Gap / Action items:** Sandboxing is low-priority unless the system begins generating or executing code. No action required now.

## Section 4: Memory, State Management & File System

### 4.1 State Management

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 4.1.1 | Current state of each workflow run tracked in persistent store | ⚠️ | Session state tracked; no explicit phase-level run state object |
| 4.1.2 | State stored externally (survives crashes) | ✅ | Firebase Firestore with in-memory write-through cache |
| 4.1.3 | Each harness run has a unique ID linking all phases and outputs | ❌ | `session_id` links turns but a single turn's phases have no shared run ID |
| 4.1.4 | State transitions logged with timestamps and triggering conditions | ⚠️ | Conversation turns logged with timestamps; phase transitions within a turn are not logged |
| 4.1.5 | System can resume a failed run from the last successful phase checkpoint | ❌ | Cannot resume mid-turn; crash loses that turn's in-progress phases |

**Score: 1/3**

**Current state:** Session state is persisted reliably to Firebase at the turn level. There is no concept of a "run" distinct from a "turn." One turn triggers multiple phases (intent → extraction → matching → output) but if it crashes mid-way, there is nothing to resume from and no audit record of which phase failed.

**Ideal state:** A `harness_runs` table that records run ID, patient case ID, current phase, phase status, start/end times, and the model used at each step. Every run is fully reconstructable from stored state.

**Gap / Action items:**
- Add a `run_id` (UUID) generated at the start of each turn, passed through all phases
- Log phase start/end + status to Firebase under that `run_id`
- This unblocks audit trail, success rate tracking, and eventually checkpoint resume

### 4.2 File System & Workspace

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 4.2.1 | A workspace scoped to each individual run (not shared) | ❌ | No workspace directories; fully in-memory |
| 4.2.2 | Each phase writes its output to a file in the workspace | ❌ | Phase outputs exist only in memory and are discarded after response |
| 4.2.3 | Files named consistently and predictably | ❌ | N/A — no file-based workspace |
| 4.2.4 | The agent can read prior phase outputs from files | ❌ | Agents rely entirely on context injection; no file reads |
| 4.2.5 | Workspace files retained after completion for audit | ❌ | No audit trail at the reasoning level |
| 4.2.6 | Sensitive patient data handled per applicable regulations | ❌ | Firebase stores conversation content including patient data; no HIPAA compliance controls documented |

**Score: 0/3**

**Current state:** The system has no audit trail at the reasoning level. Firebase records what a user asked and what the system answered, but it is impossible to reconstruct — after the fact — which guideline was retrieved, what the matching engine returned, or what the planner decided for a given turn.

**Ideal state:** Each clinical case gets an isolated workspace. Every phase writes a structured output file. Files are retained for audit and are access-controlled.

**Gap / Action items:**
- The `run_id` from 4.1 is the foundation — once each phase result is logged to Firebase under that ID, you get 80% of the audit trail without a file system
- For full workspace pattern: write phase outputs to Firebase subcollections rather than flat files
- Document Firebase data handling policy and assess against HIPAA requirements

### 4.3 Memory

| # | Checklist Item | Status | Notes |
|---|---|---|---|
| 4.3.1 | Short-term memory managed explicitly, not left to context window alone | ⚠️ | Session state managed explicitly; conversation history passed in full with no pruning |
| 4.3.2 | Knowledge base in retrievable format | ✅ | OpenAI vector store for guidelines; `enriched_recommendations.json` for structured matching; Firebase + Whoosh for devices |
| 4.3.3 | Retrieval scoped to current task | ✅ | Vector retrieval uses metadata filters and MIN_SCORE=0.4; matching engine applies patient-specific criteria |
| 4.3.4 | Long-term memory (updated guidelines) has a defined update process | ❌ | 2026 AHA/ASA guidelines hardcoded; no versioning or update process defined |
| 4.3.5 | Retrieved guideline versions are logged | ❌ | No version parameter in vector retrieval; outputs are not traceable to a specific guideline version |

**Score: 1/3**

**Current state:** Knowledge bases are well-structured and retrieval is scoped correctly. The critical gap is guideline versioning — the 2026 AHA/ASA guidelines are hardcoded. When those guidelines update, there is no mechanism to version, diff, or swap them without a code deploy. A recommendation from last month cannot be audited against the guideline that was active at that time.

**Ideal state:** Guidelines are versioned and indexed. Retrieval logs the specific document and version used. The system can be audited to show exactly which guideline informed a specific recommendation.

**Gap / Action items:**
- Add a `guideline_version` field to vector store metadata and to Firebase turn records
- Define a guideline update process (who owns it, how it is reviewed, how old recommendations are re-tested)

## What the Score Actually Means

The 12/30 is not a judgment on engineering quality — the code is well-structured, the deterministic/LLM split is correct, and the base architecture is sound. The score reflects that several *production-safety layers* do not yet exist.

### Where the system is genuinely strong

- **Deterministic-first design** — rule engine, matching engine, compat evaluator are all pure Python. Most teams get this wrong.
- **Model selection by role** — Sonnet/Haiku split is correct for this workload
- **Sub-agent isolation** — fresh context per agent, orchestrator-controlled injection, structured return contract
- **Graceful decline** — UNCERTAIN status, conditional frameworks, out-of-scope rejection all work correctly

---

## Prioritised Action Items

### Critical (must address before any clinical use)

1. **Silent JSON failure in `base_agent.py`** — replace `{"raw_text": content}` fallback with an explicit retry (1–2 attempts) then hard error for clinical paths. This is a clinical safety risk.
2. **No human review gate** — add a mandatory disclaimer to every clinical response ("for decision support only; clinician must review before acting"). A full workflow gate is the eventual goal.
3. **No reasoning-level audit trail** — implement `run_id` per turn and log each phase result to Firebase. Without this, bad recommendations cannot be investigated after the fact.

### High priority (address within current sprint)

4. **Unbounded conversation history** — implement a sliding window or summarization step to prevent silent context degradation in long sessions.
5. **No guideline versioning** — add `guideline_version` metadata to vector store and Firebase logs. Define an update process.
6. **Section 5 documentation** — write incident response process, clinical validity review plan, regulatory assessment.

### Backlog

7. Phase-level run IDs and checkpoint resume
8. Per-step pass/fail logging to Firebase
9. Formal phase naming/numbering in code
10. Gate the planned path so clinical steps cannot be LLM-generated
